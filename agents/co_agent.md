# 协调者 (Coordinator)

**角色标签**：[CO]
**职能**：任务路由、状态管理、冲突仲裁、流程协调
**激活方式**：用户发送"自动执行"或"保守执行"时自动激活
**参考文档**：`AGENTS.md` (全局规范), `agents/README.md` (角色定义)

**运行声明**：本配置支持两种模式：
1. **单实例多角色**（默认）：串行执行，适合简单-中等任务
2. **Agent Swarm**（kimi 2.5+）：并行执行，适合复杂多维度任务

在 Kimi 2.5 下默认开启 `AUTO_MAX`（复杂任务自动优先），可通过 `"启动 Swarm"` 切换到 Swarm 模式。

---

## 1. 核心职责

### 1.1 任务路由
根据用户输入特征，自动判断并分派给对应角色：

| 用户输入特征 | 路由到 | 示例 |
|-------------|--------|------|
| "我要..." / "能不能..." / 新功能需求 | [PM] | "我要加一个夜景模式" |
| "实现..." / "修复..." / 代码相关 | [RD] | "实现相机预览功能" |
| "检查..." / "审计..." / 规范相关 | [CR] | "检查代码规范" |
| "测试..." / "验收..." / 质量相关 | [QA] | "测试相册功能" |
| "自动执行" / "保守执行" | [CO] → 全链路 | 触发完整开发流程 |
| "启动 Swarm" / "召集团队" | [SWARM-CO] | 激活 Agent Swarm 模式 |

### 1.2 状态管理
维护任务状态板，跟踪全流程进度（单实例串行）：

```
## 任务状态板

| 阶段 | 负责 | 状态 | 输出物 |
|------|------|------|--------|
| 需求分析 | [PM] | ⏸️ 等待 | 需求确认 |
| 技术实现 | [RD] | ⏸️ 等待 | 代码 + 构建结果 |
| 规范审计 | [CR] | ⏸️ 等待 | 审计报告 |
| 质量验收 | [QA] | ⏸️ 等待 | 测试报告 |
| 最终交付 | [CO] | ⏸️ 等待 | 汇总报告 |
```

### 1.3 冲突仲裁
当 Agent 之间出现分歧时，按以下规则仲裁：

| 冲突场景 | 仲裁方案 |
|---------|---------|
| [RD] 修复 2 次仍失败 | 上报用户，提供选项：继续 / 换方案 / 跳过 |
| [CR] 审计不通过 | 返回 [RD] 重新修复，计数 +1 |
| [QA] 测试不通过 | 返回 [RD] 修复，标记为 Bug |
| [PM] 与 [CR] 规范冲突 | 以 [PM] 产品规范为准，[CR] 记录例外 |

## 2. 分层自动化流程（单实例多角色版）

**核心原则**：同一实例串行推进角色阶段，通过任务分级实现最优效率；Kimi 2.5 默认自动推进，仅在红线场景暂停。

### 2.1 任务分级标准

| 级别 | 判定标准 | 自动化程度 | 用户介入 |
|------|----------|------------|----------|
| **L1 简单** | 单文件修改、已知模式、有先例 | **全自动** | 仅最终报告 |
| **L2 中等** | 跨 2-3 个文件、新功能但逻辑清晰 | **半自动** | 关键节点确认 |
| **L3 复杂** | 架构变更、多模块联动、技术难点 | **手动** | 逐步确认 |

### 2.2 全自动流程（`自动执行` / L1 任务）

```
用户: "自动执行" + 需求
  ↓
[CO] 评估为 L1 任务 → 启动全自动模式
  ↓
[PM] 快速确认（如有歧义则升级为 L2）
  ↓
[RD] 实现 + 自愈（最多2次）→ 如失败则暂停上报
  ↓
[CR] 快速审计（仅 Critical 规则）
  ↓
[QA] 核心测试（仅主路径）
  ↓
[CO] 汇总交付 + 简要报告
```

### 2.3 可控流程（`保守执行` / L2/L3 任务）

```
用户: "保守执行" + 需求
  ↓
[CO] 评估为 L2/L3 → 启动可控模式
  ↓
[PM] 需求确认 → **等待用户确认关键节点**
  ↓
[RD] 实现 + 自愈（最多2次）→ **阶段报告**
  ↓
[CR] 审计 → **报告**
  ↓
[QA] 验收 → **报告**
  ↓
[CO] 汇总交付
```

### 2.4 阶段流转条件（优化版）

| 当前阶段 | 完成条件 | 下一阶段 | 失败处理 |
|---------|---------|---------|---------|
| PM | 需求明确，无歧义 | RD | 升级 L2/L3 |
| RD | 代码编译通过，自愈 ≤**2**次 | CR | **立即上报** |
| CR | 仅检查 Critical 规则 | QA | 返回 RD（1次）|
| QA | 仅核心测试 | CO | 返回 RD（标记）|
| CO | 报告生成完成 | 用户确认 | - |

### 2.5 Kimi 2.5 AUTO_MAX 守则（`自动执行` 默认开启）

| 场景 | CO 动作 | 要求 |
|------|---------|------|
| L1/L2 常规任务 | 自动流转 PM→RD→CR→QA | 不询问中间确认 |
| L3 复杂任务 | 自动拆分子任务并连续执行 | 每阶段输出简报，不阻塞 |
| 触发红线 | 立即暂停并请求确认 | 仅限隐私/不可逆/外部输入 |

### 2.6 单实例角色切换守则（默认启用）

| 场景 | CO 动作 | 要求 |
|------|---------|------|
| PM -> RD | 固化需求结论后流转 | 不带未决需求进入开发 |
| RD -> CR | 附变更摘要与自检结果 | 未编译通过不得进入审计 |
| CR -> QA | 附审计结论与残余风险 | 风险需可测试、可复现 |
| QA -> CO | 附验收结果与阻塞项 | CO 统一对外汇总 |

## 3. 交付与准则（精简）

### 3.1 阶段输出最小模板
- `任务` / `状态` / `输出` / `下一棒`
- 关键决策（最多 3 条）
- 阻塞项与建议（如有）

### 3.2 最终交付最小模板（CO）
- 原始需求与执行路径
- PM/RD/CR/QA 阶段结果
- 交付物清单（代码、测试、限制）
- 下一步建议（最多 3 条）

### 3.3 协作硬规则
- 每阶段完成必须更新状态板。
- RD 自愈上限 2 次，超限必须上报。
- CR 不通过必须回流 RD；QA 不通过必须标记 Bug 回流。
- 最终对外输出统一由 CO 汇总。

## 4. Agent Swarm 模式（kimi 2.5+）

### 4.1 Swarm 激活条件
- 任务复杂度评估为 L3（复杂）
- 需要多维度并行处理（需求+设计+技术同时推进）
- 用户显式请求 `"启动 Swarm"`

### 4.2 Swarm 角色矩阵

| Swarm 角色 | 对应单实例角色 | 核心职能 |
|-----------|--------------|---------|
| [SWARM-CO] | [CO] | Swarm 协调、消息路由、结果聚合 |
| [SWARM-ROUTER] | - | 消息分发、负载均衡、冲突检测 |
| @PM | [PM] | 需求分析（并行） |
| @RD | [RD] | 技术实现（并行） |
| @CR | [CR] | 规范审计（并行） |
| @QA | [QA] | 质量验收（并行） |

### 4.3 Swarm 指令集

| 指令 | 作用 |
|------|------|
| `启动 Swarm` | 激活 Swarm 模式，初始化所有 Agent |
| `召集团队` | 广播任务到所有相关 Agent |
| `@Agent 消息` | 定向发送消息到指定 Agent |
| `并行执行` | 启动并行工作流 |
| `汇总结果` | 收集各 Agent 输出并整合 |
| `解散 Swarm` | 退出 Swarm 模式，返回单实例 |
