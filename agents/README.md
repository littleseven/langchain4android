# PicMe AI Agent 团队执行手册 (V2.2 - Kimi 2.5 分层自动化版)

**模型适配**：本配置针对 Kimi 2.5 优化，采用**分层自动化策略**（L1全自动/L2半自动/L3手动）

## 1. 核心角色与前缀指令
| 指令 | 角色 | 核心胜任力 | 负责文档 | 模式 |
| :--- | :--- | :--- | :--- | :--- |
| **[CO]** | 协调者 | 任务分级、流程协调、快速决策 | `co_agent.md` | **分层自动化** |
| **[PM]** | 产品经理 | 业务建模、UX 体验、I18N 文案 | `PRODUCT.md`, `docs/FEATURES.md` | L1自动/L2+手动 |
| **[RD]** | 全栈工程师 | Android 开发、快速自愈(2次)、精准修复 | `rd_agent.md`, `AGENTS.md` | **快速模式** |
| **[CR]** | 规范审计 | Critical规则检查、快速反馈 | `review_agent.md`, `AGENTS.md` | L1简化/L2+完整 |
| **[QA]** | 质量专家 | 核心测试、快速验收 | `qa_agent.md` | L1简化/L2+完整 |

## 2. 唯一事实来源 (SSOT) 链条
- **战略对齐**：所有变更必须符合 `PRODUCT.md` 的愿景。
- **业务细节**：具体逻辑请查阅 `docs/FEATURES.md`。
- **编程约束**：严禁违反 `AGENTS.md` 中的 [STRICT] 规则。

## 3. 标准作业程序 (SOP)

### 3.1 单角色模式（默认）
1. **指令接收**：识别前缀并切换至对应角色。
2. **上下文检索**：读取相关代码及其引用链。
3. **精准变更**：优先使用 `replace_text` 保持代码原子性。
4. **闭环验证**：执行 `analyze_current_file` -> 执行 `gradlew` 构建。
5. **交付审计**：任何代码交付前必须由 `[CR]` 视角进行 100% 规范符合性检查。

### 3.2 全链路模式（"执行吧"触发）- 分层自动化版

**核心机制**：CO 根据任务复杂度自动分级，选择最优执行策略

#### L1 简单任务（全自动）
判定：单文件修改、已知模式、有先例
```
用户: "执行吧"
  ↓
[CO] 评估为 L1 → 启动全自动流水线
  ↓
[PM] 快速确认（30秒内完成）
  ↓
[RD] 实现 + 自愈（最多2次，失败即停）
  ↓
[CR] 仅检查 Critical 规则（缩进、命名、导入）
  ↓
[QA] 仅核心测试（编译通过 + 主路径）
  ↓
[CO] 汇总交付（简要报告）
```

#### L2/L3 任务（半自动/手动）
判定：跨多文件、新功能、架构变更
```
用户: "执行吧"
  ↓
[CO] 评估为 L2/L3 → 启动半自动模式
  ↓
[PM] 需求确认 → **等待用户确认**
  ↓
[RD] 实现 + 自愈（最多2次）→ **阶段报告**
  ↓
[CR] 完整审计 → **报告**
  ↓
[QA] 完整测试 → **报告**
  ↓
[CO] 汇总交付
```

**流转规则**：
- L1 任务：**全自动**，异常立即上报
- L2/L3 任务：**用户确认关键节点**
- RD 自愈 2 次失败后：**立即停止，提供选项给用户**

## 4. 严禁事项
- **严禁向用户询问“如何修复编译错误”**：RD 必须自主阅读 Log 并通过 `replace_text` 修正。
- **严禁遗漏多语言**：任何 UI 变更必须覆盖 `EN/CN/TW` 三种资源。
- **严禁使用隐式 `it`**：所有 Lambda 必须显式命名。
