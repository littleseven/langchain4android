# PicMe 产品定义与路线图

> **试验性应用** | 以 AI Agent 对话为核心，以相册+图像编辑为技术试验场  
> **版本**：3.0（觅影相册）  
> **状态**：生效中  
**最后更新**：2026-06-19
**维护者**：PM Agent（产品定义）+ RD Agent（技术实现）
**实验状态**：进行中 · Phase 4 架构升级（本地/远程推理协议分离 + langchain4j 标准化 + DeepSeek 适配 · 产品重心迁移至相册+图片编辑）

> **2026-06-17 IM 远程控制产品线新增**：新增 IM 远程控制产品线，通过飞书等 IM + LLM 实现 App 远程控制。智能相册功能全量规划完成（智能分类、相册管理、AI 编辑进阶、视频管理等）。

> **2026-06-19 架构升级（ADR-005）**：本地/远程推理协议正式分离。本地模型使用精简自定义 JSON 数组协议；远程模型使用标准 OpenAI Chat Completions API 协议（含原生 tool_calls、流式、多轮对话）。远程链路引入 langchain4j 1.13.0 标准化，`LangChain4jOpenAiClient` 使用 `OpenAiChatModel` 消费标准 OpenAI 协议。DeepSeek 适配完成（禁用 thinking、strict 模式兼容）。同时产品重心从相机迁移至相册与图片编辑。

PicMe 不是面向市场的商业化产品，而是一个技术探索实验场。我们的核心命题是：**当端侧 AI Agent 成为应用的中枢神经系统时，传统的 App 架构和交互范式将如何演进？**

相册与图像编辑是这个命题的核心试验场景——它们天然适合与端侧智能结合（本地 AI 修图、隐私安全的图像分析、对话式参数调节），同时提供了丰富的端侧模型交互验证路径（对话是否自然、推理是否可靠、响应是否及时）。

## 1. 核心命题

### 1.1 传统 GUI vs Agent 驱动

| 维度 | 传统 GUI App | Agent 驱动 App |
|------|--------------|----------------|
| 交互入口 | 按钮、菜单、层级导航 | 自然语言对话 + 意图识别 |
| 状态反馈 | 视觉组件状态变化 | 自然语言解释 + 渐进式确认 |
| 功能发现 | 用户学习界面布局 | Agent 主动推荐 + 上下文感知 |
| 错误处理 | 弹窗、Toast 提示 | 对话式澄清 + 智能容错 |
| 架构核心 | ViewModel + UI State | Agent Runtime + Capability Registry |

PicMe 的实验目标是探索**右侧范式的工程可行性**。

### 1.2 技术试验矩阵

```
                    技术深度
                       ▲
                       │
    OpenGL ES/EGL    ──┼──  实时美颜渲染管线
    多引擎人脸检测   ──┼──  MediaPipe/NCNN/MNN 统一抽象
    帧同步系统       ──┼──  检测-渲染时序对齐
                       │
    ═══════════════════╪═══════════════════
                       │
    Qwen3.5-2B/MNN ──┼──  端侧 LLM 推理（轻量对话+指令路由）
    Capability系统   ──┼──  自然语言→设备命令映射
    对话记忆         ──┼──  多轮上下文维护
                       │
                       ▼
                    Agent 框架
         ◄─────────────────────────►
         工程可行性                    交互自然度
```

**当前聚焦**：从「相机工具」向「AI 相册助手」转型，以相册首页为核心入口，AI 对话、相机、模型中心作为二级能力被底部悬浮 Tab 聚合，设置入口位于顶部栏最右侧，相册/编辑作为核心能力被 Agent 调度。

**关键架构更新（2026-06-19）**：
1. **本地/远程推理协议彻底分离**（ADR-005）：本地使用自定义 JSON 数组协议（method + args 平铺），远程使用标准 OpenAI Chat Completions API 协议（原生 tool_calls·流式·多轮对话）。两条链路完全独立，无共享路由逻辑。
2. **LangChain4j 标准化**：远程推理链路引入 langchain4j 1.13.0，`LangChain4jOpenAiClient` 使用 `OpenAiChatModel` 消费标准 OpenAI 协议。`UnifiedRemoteClient` 根据协议自动路由（OPENAI → langchain4j / CLAUDE → Retrofit）。
3. **DeepSeek 适配**：API 请求自动禁用 thinking 模式；ToolSpec 自动添加 `additionalProperties: false` 兼容 strict 模式；`tool_choice: REQUIRED` 正确映射为 `"required"`。
4. **冗余代码清理**：移除 InferenceRouter、ToolCallingChatLanguageModel、ToolCallingOutputParser、ToolPromptBuilder、ToolCallingMode、ToolCallingConfig、AdaptiveStrategySelector、ToolOrchestrator 等 ~1500 行代码。
5. **产品重心迁移**：相册（Gallery）和图片编辑（Editor）从「能力验证」升级为「核心产品」，相机降级为「辅助入口」。

**关键认知更新（2026-06-12，保留）**：
1. 端侧 Qwen3.5-2B 模型验证结果表明，其能力边界为「指令路由器+轻量对话」而非「深度推理引擎」。Function Calling、简单指令翻译、基础闲聊胜任，但复杂推理、多步规划、上下文推理超出能力范围。
2. 产品策略据此调整：端侧专注 L1/L2（缓存+单条/简单批量 FC+基础对话），复杂推理明确上云或降级为规则引擎。
3. **首页转型**：相册页作为应用默认首页，聊天页、相机页、模型中心页作为图标入口从相册底部悬浮 Tab 进入（无文字标签）；设置入口位于顶部栏最右侧，模型中心同时保留在设置页 AI 助手卡片第一项。相机降级为辅助入口，相册+编辑升级为核心能力，AI 对话作为核心二级能力。

## 2. 产品形态

### 2.1 核心能力

**相册首页层（应用默认入口）**
- 相册网格/时间轴浏览作为打开 App 后的首屏
- 底部悬浮 Tab 聚合 Camera / Chat / Model Center 三个图标入口（无文字标签）
- 设置入口统一放在顶部栏最右侧；模型中心同时保留在设置页 AI 助手卡片第一项
- 保留语音 Agent 入口，支持从相册直接通过自然语言调度能力
- 智能分类、搜索、去重、TAG 扫描等相册能力首页可达

**AI 对话层（二级页）**
- 用户通过自然语言与 AI 进行多轮对话
- 支持本地模型（Qwen3.5-2B）/ 远程模型（DeepSeek 等）实时切换
- 对话历史持久化，支持跨会话查看历史记录
- 支持发送图片进行 AI 分析/编辑建议
- 对话式反馈：操作确认、错误澄清、建议推荐、结果展示
- 从相册首页 plus 菜单进入，顶部栏提供返回相册按钮

**IM 远程控制层（融合入口 - 新增）**
- 飞书机器人接入：通过 IM 消息执行相册浏览/编辑/管理
- LLM 语义解析：将 IM 自然语言指令转为设备可执行命令
- 设备直连飞书：设备端与飞书平台建立 WebSocket 长连接，无需云端中转
- 多设备管理：同一账号绑定多台设备，支持@指定
- 结果回传：编辑后图片直接通过飞书返回

**相册+编辑层（核心产品）**
- 相册浏览：时间轴 + 缩略图，120fps 滑动
- 静态图美颜编辑：复用相机美颜管线，GPU 离屏渲染（PhotoProcessorImpl）
- AI 一键优化：智能识别场景，一键推荐参数
- 精准局部美颜：左眼/右眼/左脸/右脸独立调节
- **对话式编辑**：通过聊天指令调节编辑参数（"磨皮再强一点"）
- **批量处理**：多图批量应用同一套美颜/滤镜参数
- **智能搜索**：按日期/地点/内容自然语言搜索相册

**相机能力层（辅助入口）**
- 实时美颜（磨皮、美白、瘦脸、大眼、唇色、腮红）
- 风格滤镜（色调调整 + GPU 特效）
- 拍照/录像（GPU 离屏处理确保预览/输出一致）
- 从相册首页 plus 菜单或聊天页快捷入口进入，拍照后返回进入页
- **AI 相机功能降级为 P2**：语音拍照、实时调节等 Agent 功能短期不新增投入

### 2.2 体验红线

| 红线 | 定义 | 验证方式 |
|------|------|----------|
| **Agent 响应** | < 800ms（本地 L2 推理）/ < 1.5s（远程 L3/L4 推理） | 日志埋点 |
| **交互反馈** | < 100ms（UI 响应） | 人工感知测试 |
| **对话首字延迟** | < 500ms（本地）/ < 1s（远程） | 日志埋点 |
| **快门延迟** | < 50ms | 高速摄像 |
| **隐私绝对** | 敏感数据强制本地推理，100% 端侧 | 权限清单审查 |
| **IM 命令响应（新增）** | < 3s（IM 发出到 Bot 回复） | 端到端计时 |
| **IM 图片处理（新增）** | 单张编辑 < 5s @1080p | 计时 |

## 3. 非目标（明确不做）

为了保持试验聚焦，以下方向**明确不做**：

1. **云端服务**：不接入云端存储/同步；云 LLM 仅作为非敏感指令的降级兜底，敏感数据（人脸/对话/图片）100% 端侧
2. **社交功能**：没有分享、没有社区、没有好友系统
3. **商业变现**：无广告、无订阅、无内购
4. **跨平台**：专注 Android，不考虑 iOS 或其他平台
5. **通用 Agent**：不做跨应用调用、不做系统级自动化，仅聚焦相册+编辑+对话+IM远程控制场景
6. **复杂推理端侧化**：避免在端侧小模型上测试不可靠的复杂推理能力；明确端侧边界为 L1/L2（缓存+单条/简单批量指令+基础闲聊），复杂推理通过远程模型或规则引擎处理

## 4. 架构原则

### 4.1 Agent 中心化

```
User Input (Voice/Text/Image)
    ↓
ChatScreen (首页 - features/chat/)
    ├─ ChatViewModel (对话状态管理)
    ├─ MessageRepository (对话持久化)
    └─ ModelSelector (本地/远程切换)
        ↓
AgentOrchestrator (agent-core/)
    ├── LOCAL: LocalInferencePipeline
    │   ├── LocalLlmEngine (Qwen3.5-2B-MNN)
    │   └── 自定义 JSON 数组协议
    ├── REMOTE: RemoteInferencePipeline
    │   ├── RemoteOrchestrator (OpenAI 标准 API)
    │   └── Chat Completions · tool_calls · 流式
    ├── MemoryManager (对话上下文 + 持久化)
    ├── PrivacyGuard (隐私分级)
    └── CapabilityRegistry (能力路由)
        ↓
    RemoteInferenceCapability (features/agent/remote/) - IM远程融合
    CameraCapability (features/camera/) - 辅助入口
    GalleryCapability (features/gallery/) - 核心能力
    EditorCapability (features/editor/) - 核心能力
    SettingsCapability (features/settings/)
    NavigationCapability (global)
        ↓
    UI Feedback (Compose)
    IM Feedback (飞书卡片/消息)
```

**关键决策**：Agent 层是"唯一可信源"，UI 是对 Agent 状态的反映。IM 远程控制是对 Agent 能力的延伸，同一套 Capability 同时服务 App 内对话和 IM 远程请求。

### 4.2 能力可插拔

每个 `Capability` 是独立模块：
- 声明输入/输出 Schema
- 自包含错误处理
- 可被 Agent 动态发现和调用

新增能力只需：实现 `Capability` 接口 → 注册到 `CapabilityRegistry` → Agent 自动获得该能力。

### 4.3 端侧优先

- **LLM**：当前默认 Qwen3.5-2B（Qwen3-1.7B 已升级），通过 MNN-LLM 本地运行
- 人脸检测：MediaPipe Face Mesh / MNN / NCNN 端侧模型（106 点统一输出）
- OCR：ML Kit 端侧识别
- 图像编辑：100% 端侧 GPU 处理
- **隐私敏感数据强制本地处理**是硬性约束

### 5.3 长期问题

这些问题没有标准答案，是 PicMe 持续探索的方向：

1. **Agent 的可解释性**：如何让 Agent 的决策过程对用户透明？
2. **错误恢复的优雅性**：当 Agent 理解失败时，如何引导用户而非简单报错？
3. **交互与自动化的边界**：哪些操作应该由 Agent 自动完成，哪些需要用户确认？
4. **个性化与隐私的平衡**：如何在端侧实现个性化学习而不侵犯隐私？

## 6. 2026 路线图

### 6.1 战略总览

PicMe 以技术探索与能力验证为核心目标，三条产品线并行推进：

| 产品线 | 定位 | 核心价值 | 研究目标 |
|--------|------|----------|----------|
| **AI 对话** | **核心入口** | 自然语言交互 + 模型切换 + 对话持久化 | 端侧模型对话体验、人机交互范式验证 |
| **相册及编辑** | **核心能力** | 本地 AI 修图 + 精准局部控制 + 隐私安全 | 美颜算法离线效果评估、编辑体验研究 |
| **IM 远程控制** | **融合入口** | IM + LLM 远程操控 + 飞书机器人接入 | 跨平台远程 App 控制、IM 即时交付图片编辑 |
| **拍照** | 能力验证基线 | 零延迟 + 自然美颜 | 实时渲染管线、人脸关键点驱动美颜 |

```
AI 对话（核心入口）→ 相册+编辑（核心能力）→ IM远程控制（融合入口）→ 拍照（能力基线）→ 视频（极限挑战）

                    大美丽引擎统一支撑（美颜/滤镜/妆容/关键点）
```

## 8. 风险与应对

| 风险 | 可能性 | 影响 | 应对策略 |
|------|--------|------|----------|
| **聊天首页体验不佳** | 高 | 用户不接受对话为核心入口 | 保留相机快捷入口作为 fallback；A/B 测试验证留存率 |
| **端侧模型对话质量差** | 高 | Qwen3.5-2B 对话能力不达预期 | 默认远程模型；端侧仅作为离线 fallback；持续优化提示词 |
| 帧同步技术延期 | 中 | 视频美颜录制受阻 | 预留 2 周 buffer；如延期先上视频后补帧同步 |
| 视频录制性能不达标 | 中 | 1080p 录制丢帧 | 先保证 720p@30fps，再优化 1080p |
| 智能消除效果差 | 中 | 编辑核心功能口碑崩 | 初期标注「Beta」，逐步迭代模型 |
| 资源分散 | 中 | 四线并行导致质量下降 | 按 Phase 聚焦，单阶段最多两条线并行；对话首页绝对优先 |
| 技术深度不足 | 中 | 研究成果缺乏竞争力 | 定期对标行业最新论文与开源方案，保持技术前沿性 |
