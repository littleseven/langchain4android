# PicMe 产品定义

> **试验性应用** | 以 AI Agent 交互框架为核心，以美颜相机为技术试验场

PicMe 不是面向市场的商业化产品，而是一个技术探索实验场。我们的核心命题是：**当端侧 AI Agent 成为应用的中枢神经系统时，传统的 App 架构和交互范式将如何演进？**

美颜相机只是这个命题的一个具体试验场景——它提供了丰富的实时计算需求（OpenGL 渲染、人脸检测、多路数据同步），以及直观的用户价值验证路径（拍得美不美、交互自不自然）。

---

## 1. 核心命题

### 1.1 传统 GUI vs Agent 驱动

| 维度 | 传统 GUI App | Agent 驱动 App |
|------|--------------|----------------|
| 交互入口 | 按钮、菜单、层级导航 | 自然语言对话 + 意图识别 |
| 状态反馈 | 视觉组件状态变化 | 自然语言解释 + 渐进式确认 |
| 功能发现 | 用户学习界面布局 | Agent 主动推荐 + 上下文感知 |
| 错误处理 | 弹窗、Toast 提示 | 对话式澄清 + 智能容错 |
| 架构核心 | ViewModel + UI State | Agent Runtime + Capability Registry |

PicMe 的实验目标是探索**右侧范式的工程可行性**。

### 1.2 技术试验矩阵

```
                    技术深度
                       ▲
                       │
    OpenGL ES/EGL    ──┼──  实时美颜渲染管线
    多引擎人脸检测   ──┼──  ONNX/NCNN/MNN/MediaPipe 统一抽象
    帧同步系统       ──┼──  检测-渲染时序对齐
                       │
    ═══════════════════╪═══════════════════
                       │
    Qwen3-0.6B/MNN   ──┼──  端侧 LLM 推理
    Capability系统   ──┼──  自然语言→设备命令映射
    对话记忆         ──┼──  多轮上下文维护
                       │
                       ▼
                    Agent 框架
         ◄─────────────────────────►
         工程可行性                    交互自然度
```

**当前聚焦**：左下象限的「工程可行性」验证，即确保 Agent 系统能在端侧稳定、快速、隐私安全地运行。

## 2. 产品形态

### 2.1 核心能力

**Agent 交互层**
- 用户通过自然语言与相机交互
- 支持意图：美颜调节、滤镜切换、相机控制、参数查询、闲聊
- 对话式反馈：操作确认、错误澄清、建议推荐

**相机能力层**
- 实时美颜（磨皮、美白、瘦脸、大眼、唇色、腮红）
- 风格滤镜（色调调整 + GPU 特效）
- 拍照/录像（GPU 离屏处理确保预览/输出一致）

### 2.2 体验红线

| 红线 | 定义 | 验证方式 |
|------|------|----------|
| **Agent 响应** | < 500ms（本地 LLM 推理） | 日志埋点 |
| **交互反馈** | < 100ms（UI 响应） | 人工感知测试 |
| **快门延迟** | < 50ms | 高速摄像 |
| **隐私绝对** | 零网络权限，100% 端侧 | 权限清单审查 |

## 3. 非目标（明确不做）

为了保持试验聚焦，以下方向**明确不做**：

1. **云端服务**：不接入任何云端 API，包括云 LLM、云存储、云同步
2. **社交功能**：没有分享、没有社区、没有好友系统
3. **商业变现**：无广告、无订阅、无内购
4. **跨平台**：专注 Android，不考虑 iOS 或其他平台
5. **通用 Agent**：不做跨应用调用、不做系统级自动化，仅聚焦相机场景

## 4. 架构原则

### 4.1 Agent 中心化

```
User Input (Voice/Text)
    ↓
AgentOrchestrator (domain/agent/)
    ├─ LocalLlmEngine (Qwen3-0.6B-MNN)
    ├─ MemoryManager (对话上下文)
    ├─ PrivacyGuard (隐私分级)
    └─ CapabilityRegistry (能力路由)
        ↓
    CameraCapability (features/camera/)
    BeautyCapability (beauty-engine/api/)
    SystemCapability (framework/)
        ↓
    UI Feedback (Compose)
```

**关键决策**：Agent 层是"唯一可信源"，UI 是对 Agent 状态的反映，而非直接控制业务逻辑。

### 4.2 能力可插拔

每个 `Capability` 是独立模块：
- 声明输入/输出 Schema
- 自包含错误处理
- 可被 Agent 动态发现和调用

新增能力只需：实现 `Capability` 接口 → 注册到 `CapabilityRegistry` → Agent 自动获得该能力。

### 4.3 端侧优先

- LLM：Qwen3-0.6B 通过 MNN-LLM 本地运行
- 人脸检测：InsightFace/MediaPipe 端侧模型
- OCR：ML Kit 端侧识别
- **零网络权限**是硬性约束

## 5. 当前状态与下一步

### 5.1 已验证

| 能力 | 状态 | 备注 |
|------|------|------|
| Agent 基础架构 | ✅ | Orchestrator + LLM Engine + Capability Registry |
| 自然语言→命令 | ✅ | Qwen3-0.6B 解析意图，映射到设备操作 |
| 实时美颜 | ✅ | OpenGL ES 管线，支持磨皮/美白/瘦脸/大眼/唇色/腮红 |
| GPU 拍照 | ✅ | 离屏渲染，预览/输出一致性 |
| 多引擎人脸检测 | ✅ | InsightFace(ONNX) 为主，MediaPipe 备选 |
| 对话记忆 | ✅ | 多轮上下文维护 |

### 5.2 待验证

| 能力 | 优先级 | 验证目标 |
|------|--------|----------|
| 复杂意图理解 | P0 | 多参数同时调节（"瘦脸 30，美白 50"）|
| 上下文推理 | P1 | 基于对话历史的隐式引用（"再亮一点"）|
| 主动建议 | P2 | Agent 根据场景主动推荐（"光线较暗，是否开启夜景？"）|
| 帧同步美妆 | P1 | 解决快速移动时的妆容甩飞问题 |

### 5.3 长期问题

这些问题没有标准答案，是 PicMe 持续探索的方向：

1. **Agent 的可解释性**：如何让 Agent 的决策过程对用户透明？
2. **错误恢复的优雅性**：当 Agent 理解失败时，如何引导用户而非简单报错？
3. **交互与自动化的边界**：哪些操作应该由 Agent 自动完成，哪些需要用户确认？
4. **个性化与隐私的平衡**：如何在端侧实现个性化学习而不侵犯隐私？

## 6. 验收标准

### 6.1 Agent 体验

- [ ] 用户可以用自然语言完成 80% 的常用操作
- [ ] 意图识别准确率 > 90%（在定义的意图范围内）
- [ ] 对话式错误恢复成功率 > 70%

### 6.2 技术性能

- [ ] LLM 首 token 延迟 < 500ms（骁龙 8 Gen2 基准）
- [ ] 端到端命令执行延迟 < 1s
- [ ] 应用包体积 < 150MB（含模型）

### 6.3 工程健康

- [ ] Capability 系统支持热插拔（新增能力无需修改 Agent 核心）
- [ ] 单测覆盖率 > 60%（Agent 核心逻辑）
- [ ] 零崩溃率（已发布版本）
