# 大美丽 106 与 InsightFace 106 映射说明

## 1. 文档目的

本文用于说明 PicMe 工程中"大美丽 106"与 `InsightFace 106` 之间的关系、映射规则、实现差异。

本文重点回答以下问题：

- 大美丽 106 与 InsightFace 106 的完整映射关系
- 两套 106 的点序差异
- 如何通过映射表实现精准转换

## 2. 结论

在当前工程里，大美丽 106 与 `InsightFace 106` 的关系是：**所有 106 个点都需要重排映射，不存在 index -> same index 的直通点**。

结论如下：

- **所有 106 个点均需要重排映射**
- 不存在 `BigBeauty[i] == InsightFace[i]` 的情况
- 如果映射表中出现 `i -> i` 的情况，可以直接判定为错误
- 轮廓点（0-32）和非轮廓点（33-105）均需要逐点位重新映射
- 真正复杂的映射发生在 `MediaPipe 468 -> 统一 106` 和 `InsightFace 106 -> 统一 106` 两步

## 3. 概念定义

### 3.1 大美丽 106

"大美丽 106"不是某个独立模型的原生输出名，而是工程里供美型、妆容、调试 UI、Warp 参数转换统一消费的一套 106 点标准拓扑。

这套拓扑的语义基于项目内部约定的统一 106 标准，并与火山/行业通用 106 点语义保持一致。

### 3.2 InsightFace 106

`InsightFace 106` 指 `2d106det.onnx` 直接输出的 106 个人脸关键点。

在当前项目中，这 106 点的**所有点都需要重排映射到统一 106 标准**，不存在同索引直通的情况。

**重要**：根据实际图片对比（standard_106_points.png vs insightface_106_points.png），InsightFace 106 与火山引擎 106 的点序完全不同，每个点都需要通过映射表转换。

InsightFace 轮廓排列特征：
- 0：下巴中心
- 1-16：左半边脸（从下巴左侧逆时针到左鬓角）
- 17-32：右半边脸（从右鬓角顺时针到下巴右侧）

火山引擎轮廓排列特征：
- 0-15：左半边脸（从左鬓角顺时针到下巴左侧）
- 16：下巴中心
- 17-32：右半边脸（从下巴右侧顺时针到右鬓角）

**非轮廓点差异**：InsightFace 的眉毛、眼睛、鼻子、嘴巴等区域索引排列与火山引擎标准完全不同，需要逐点位重新映射。

## 5. 映射规则

### 5.1 索引规则

大美丽统一 106 与 `InsightFace 106` 的映射规则：**所有 106 个点均需要重排映射**。

**重要原则**：
- 不存在 `index -> same index` 的直通点
- 如果映射表中出现 `i -> i` 的情况，可以直接判定为错误
- 每个点都需要通过 `FULL_REMAP` 表进行转换

### 5.2 语义规则

**所有点都需要重排映射后语义才对齐**

不存在同索引语义相同的情况，每个点都需要通过映射表转换。

典型点位示例（映射后）：

- `BigBeauty[0]（左鬓角）== InsightFace[16]（左鬓角）`
- `BigBeauty[16]（下巴中心）== InsightFace[0]（下巴中心）`
- `BigBeauty[43]（眉心）== InsightFace[74]（眉心）`
- `BigBeauty[49]（鼻尖中心）== InsightFace[79]（鼻尖中心）`
- `BigBeauty[87]（上唇中心）== InsightFace[67]（上唇中心）`
- `BigBeauty[104]（右瞳孔）== InsightFace[17]（右瞳孔）`

## 6. 真正的映射发生在哪里

### 6.1 MediaPipe 路径：需要做 `468 -> 106`

`MediaPipe Face Landmarker` 输出的是 468 点，因此必须先映射到统一 106。

这一步的特点是：

- 需要选取语义等价点
- 部分区域需要插值补点
- 需要保证左右对称关系
- 轮廓区域需要整理为统一 33 点开放曲线

因此，`MediaPipe 468 -> 106` 是一个真实存在且相对复杂的映射过程。

### 6.2 InsightFace 路径：需要完整 106 点重排

`InsightFace` 的 `2d106det.onnx` 本身直接输出 106 点。在当前项目约定里：

- **所有点（0-105）**：与统一 106 拓扑完全不同，需要通过 `FULL_REMAP` 完整重排映射

映射流程：

1. 通过 `FULL_REMAP` 表将 InsightFace 106 点重排到统一 106 顺序
2. 将裁剪图上的点位映回原图
3. 归一化到 `0..1`
4. 按当前输入链路处理镜像规则

## 7. 坐标空间与镜像规则

统一 106 的消费链路要求点位最终进入同一坐标空间。

对 `InsightFace` 路径而言，模型输出通常会经历以下步骤：

1. 模型输出裁剪图坐标
2. 通过 `inverseTransform` 映回原图
3. 归一化为 `0..1`
4. 在特定场景下执行 `x` 镜像

需要特别注意：

- 实时相机预览与静态图页面的镜像规则不一定完全相同
- 前者通常与镜头朝向绑定
- 后者还取决于原图是否已镜像、EXIF 是否已处理、显示控件是否再次翻转

因此：

- 统一 106 的拓扑一致
- 最终显示是否左右正确，仍取决于具体输入链路的坐标系处理

## 8. 两者真正的差异

如果两套 106 点叠加时看起来不完全重合，差异通常来自以下因素：

- **映射表错误**（最高优先排查）
  - 未使用 `FULL_REMAP` 重排表
  - 重排表索引计算错误
  - 误用了 1:1 直通映射

- **检测器本身不同**
  - `MediaPipe` 是 468 点再压缩到 106
  - `InsightFace` 是模型直接输出 106

- **人脸框不同**
  - `InsightFace` 路径依赖前置找脸框再做 crop
  - crop 范围、scale、偏移都会影响最终点位

- **反变换不同**
  - `inverseTransform` 的准确性决定了点位能否正确映回原图

- **镜像策略不同**
  - 相机预览、自拍图、静态图展示可能存在不同镜像语义

- **UI 显示缩放不同**
  - 原图坐标与界面坐标之间如果换算不一致，也会表现为"点位偏了"

因此，看见偏差不等于映射错了。很多情况下，问题更可能来自：

- 映射表错误或未使用映射表
- 检测误差
- crop 误差
- 坐标反算误差
- 静态图显示空间与原图空间不一致
- 镜像方向未统一

## 9. 排查建议

后续如果出现"InsightFace 106 显示不对"这类问题，建议按以下优先级排查：

1. **先查映射表**
   - 是否使用了 `FULL_REMAP` 重排表
   - 重排表索引是否正确
   - 是否存在 i -> i 的错误直通映射

2. 再查坐标空间
   - 原图坐标、归一化坐标、UI 坐标是否一致

3. 再查镜像逻辑
   - 是否把前置相机镜像规则错误复用到静态图

4. 再查 crop 与反变换
   - 检测框扩大比例、裁剪矩阵、映回原图是否准确

经验上：

- 如果表现为整体左右反了或上下颠倒，像映射表错误
- 如果表现为嘴角跑到眼角这类局部语义串位，像索引顺序错误
- 如果表现为位置整体偏移、缩放不对、换图后更明显，像裁剪或 UI 映射问题
- 如果眉毛、眼睛、鼻子、嘴巴等区域全部错位，像未使用映射表或映射表完全错误

## 10. 最终结论

PicMe 当前工程中的大美丽 106 与 `InsightFace 106` 的关系是：

- **所有 106 个点均需要重排映射**
- **不存在同索引直通的情况**
- **必须通过 `FULL_REMAP` 表进行完整转换**

真正需要关注的映射步骤：

- `MediaPipe 468 -> 统一 106`（复杂映射）
- `InsightFace 106 -> 统一 106`（完整重排映射）

因此，在实际问题定位中，如果 `InsightFace` 结果看起来"点位不对"或"语义串位"，应优先从以下方向排查：

- **映射表是否正确（最高优先）**
- 镜像策略
- crop / `inverseTransform`
- 静态图显示坐标系
