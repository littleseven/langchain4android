# 大美丽 106 与 InsightFace 106 映射说明

## 1. 文档目的

本文用于说明 PicMe 工程中“大美丽 106”与 `InsightFace 106` 之间的关系、映射规则、实现差异，以及排查关键点偏差时应优先关注的问题边界。

本文重点回答以下问题：

- 大美丽 106 与 InsightFace 106 是否需要二次重排
- 两套 106 的真正“映射”发生在哪里
- 为什么两者看起来有偏差时，往往不是索引规则错误

## 2. 结论

在当前工程里，大美丽 106 与 `InsightFace 106` 不是“两套不同拓扑之间再做一次重排”的关系，而是统一收敛到同一套 106 点标准。

结论如下：

- 两者使用同一套统一 106 点拓扑标准
- 映射规则本质上是 `index -> same index` 的 1:1 映射
- `BigBeauty[i]` 和 `InsightFace[i]` 表示同一个语义点
- 当前工程中不存在 `InsightFace 106 -> BigBeauty 106` 的额外重排表
- 真正复杂的映射发生在 `MediaPipe 468 -> 统一 106` 这一步

## 3. 概念定义

### 3.1 大美丽 106

“大美丽 106”不是某个独立模型的原生输出名，而是工程里供美型、妆容、调试 UI、Warp 参数转换统一消费的一套 106 点标准拓扑。

这套拓扑的语义基于项目内部约定的统一 106 标准，并与火山/行业通用 106 点语义保持一致。

### 3.2 InsightFace 106

`InsightFace 106` 指 `2d106det.onnx` 直接输出的 106 个人脸关键点。

在当前项目中，这 106 点的输出顺序被视为与统一 106 消费链路兼容，因此可以直接复用后续处理逻辑，而不需要再做索引重排。

## 4. 代码依据

`InsightFace` 检测器已明确说明：其输出顺序与当前 106 点消费链路一致，可以直接复用统一的参数转换逻辑。

- 参考：`app/src/main/java/com/picme/features/camera/facedetect/InsightFace2D106Detector.kt`
- 参考：`app/src/main/java/com/picme/features/camera/facedetect/Face106ToWarpParams.kt`

同时，`MediaPipe` 相关文档也说明了真正复杂的步骤是 `468 -> 106` 的语义映射与插值，而不是 `106 -> 106` 的重排。

- 参考：`docs/MEDIAPIPE_468_TO_106_MAPPING_STRATEGY.md`
- 参考：`docs/VOLCANO_ENGINE_106_POINTS.md`

## 5. 映射规则

### 5.1 索引规则

大美丽统一 106 与 `InsightFace 106` 的映射规则为同索引直通：

- `0 -> 0`
- `1 -> 1`
- `2 -> 2`
- `...`
- `105 -> 105`

即：

- 不做 `reorder`
- 不做区域互换
- 不做左右半脸成组翻转
- 不存在额外的索引对照表

### 5.2 语义规则

同索引不仅编号相同，而且语义相同。典型点位示例：

- `16`：下巴中心
- `43`：眉心
- `49`：鼻尖中心
- `87`：上唇中心
- `93`：下唇中心
- `104/105`：瞳孔中心

因此可以将其理解为：

- `BigBeauty[49] == InsightFace[49]`
- `BigBeauty[87] == InsightFace[87]`
- `BigBeauty[104] == InsightFace[104]`

这些点在工程消费语义上属于同位点。

## 6. 真正的映射发生在哪里

### 6.1 MediaPipe 路径：需要做 `468 -> 106`

`MediaPipe Face Landmarker` 输出的是 468 点，因此必须先映射到统一 106。

这一步的特点是：

- 需要选取语义等价点
- 部分区域需要插值补点
- 需要保证左右对称关系
- 轮廓区域需要整理为统一 33 点开放曲线

因此，`MediaPipe 468 -> 106` 是一个真实存在且相对复杂的映射过程。

### 6.2 InsightFace 路径：直接输出 106

`InsightFace` 的 `2d106det.onnx` 本身直接输出 106 点。在当前项目约定里，它与统一 106 拓扑兼容，因此只需要：

- 将裁剪图上的点位映回原图
- 归一化到 `0..1`
- 按当前输入链路处理镜像规则

它不是重新定义另一套 106，而是直接进入同一套 106 消费链路。

## 7. 坐标空间与镜像规则

统一 106 的消费链路要求点位最终进入同一坐标空间。

对 `InsightFace` 路径而言，模型输出通常会经历以下步骤：

1. 模型输出裁剪图坐标
2. 通过 `inverseTransform` 映回原图
3. 归一化为 `0..1`
4. 在特定场景下执行 `x` 镜像

需要特别注意：

- 实时相机预览与静态图页面的镜像规则不一定完全相同
- 前者通常与镜头朝向绑定
- 后者还取决于原图是否已镜像、EXIF 是否已处理、显示控件是否再次翻转

因此：

- 统一 106 的拓扑一致
- 最终显示是否左右正确，仍取决于具体输入链路的坐标系处理

## 8. 两者真正的差异

如果两套 106 点叠加时看起来不完全重合，差异通常来自以下因素，而不是索引映射错误：

- 检测器本身不同
  - `MediaPipe` 是 468 点再压缩到 106
  - `InsightFace` 是模型直接输出 106
- 人脸框不同
  - `InsightFace` 路径依赖前置找脸框再做 crop
  - crop 范围、scale、偏移都会影响最终点位
- 反变换不同
  - `inverseTransform` 的准确性决定了点位能否正确映回原图
- 镜像策略不同
  - 相机预览、自拍图、静态图展示可能存在不同镜像语义
- UI 显示缩放不同
  - 原图坐标与界面坐标之间如果换算不一致，也会表现为“点位偏了”

因此，看见偏差不等于映射错了。很多情况下，问题更可能来自：

- 检测误差
- crop 误差
- 坐标反算误差
- 静态图显示空间与原图空间不一致
- 镜像方向未统一

## 9. 排查建议

后续如果出现“InsightFace 106 显示不对”这类问题，建议按以下优先级排查：

1. 先查坐标空间
   - 原图坐标、归一化坐标、UI 坐标是否一致
2. 再查镜像逻辑
   - 是否把前置相机镜像规则错误复用到静态图
3. 再查 crop 与反变换
   - 检测框扩大比例、裁剪矩阵、映回原图是否准确
4. 最后才怀疑索引拓扑
   - 因为当前工程约定中，106 与 106 之间没有额外重排

经验上：

- 如果表现为整张脸整体左右反了，更像镜像问题
- 如果表现为位置整体偏移、缩放不对、换图后更明显，更像裁剪或 UI 映射问题
- 如果表现为嘴角跑到眼角这类局部语义串位，才更像索引顺序问题

## 10. 最终结论

PicMe 当前工程中的大美丽 106 与 `InsightFace 106` 使用的是同一套统一 106 点拓扑标准。因此它们之间的映射规则不是重排映射，而是：

- 同索引直通
- 同语义对齐
- 同消费链路复用

真正复杂的映射只存在于：

- `MediaPipe 468 -> 统一 106`

而不是：

- `InsightFace 106 -> 统一 106`

因此，在实际问题定位中，如果 `InsightFace` 结果看起来“左右不对”或“位置偏差明显”，应优先从以下方向排查：

- 镜像策略
- crop / `inverseTransform`
- 静态图显示坐标系

而不是先怀疑 106 点拓扑映射表本身。
