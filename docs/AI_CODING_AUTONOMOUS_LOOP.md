## 2. 设计目标：五层自闭环架构

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                         Layer 5: 交付闭环 (Deliver)                          │
│    代码提交 → PR 描述生成 → 变更影响报告 → 版本打 tag → 自动发布通知         │
├─────────────────────────────────────────────────────────────────────────────┤
│                        Layer 4: 验证闭环 (Validate)                          │
│    单元测试 → 编译检查 → 设备安装 → 截屏对比 → 日志断言 → 回归测试          │
├─────────────────────────────────────────────────────────────────────────────┤
│                         Layer 3: 编码闭环 (Code)                             │
│    架构决策 → 代码生成 → 增量编译 → 错误诊断 → 自动修复 → 验证通过          │
├─────────────────────────────────────────────────────────────────────────────┤
│                        Layer 2: 规划闭环 (Plan)                              │
│    需求解析 → 影响分析 → 方案设计 → 任务拆解 → 依赖检查 → 计划确认          │
├─────────────────────────────────────────────────────────────────────────────┤
│                        Layer 1: 意图闭环 (Intent)                            │
│    用户输入 → 意图识别 → 上下文加载 → 知识检索 → 歧义消除 → 任务启动        │
└─────────────────────────────────────────────────────────────────────────────┘
```

**每层设计原则**：
- **自闭环**：该层内部形成完整的感知→决策→执行→验证循环
- **可观测**：每层都有明确的输出产物和状态指标
- **可回退**：当自动决策置信度低于阈值时，自动降级为人工确认
- **增量演进**：每层可独立优化，不阻塞其他层

## 3. 逐层设计详解

### Layer 1: 意图闭环 (Intent)

**目标**：将用户的一句自然语言转化为 AI 可执行的、无歧义的任务描述。

#### 3.1.1 意图识别矩阵

```
用户输入                    →    意图分类    →    自动加载上下文
─────────────────────────────────────────────────────────────────
"相册加个编辑功能"           →   [Feature]   →   加载 Gallery AGENTS.md
                                                        + FEATURES.md §2
                                                        + 相关源码
"拍照后处理变慢了"           →   [BugFix]    →   加载 PhotoProcessor 源码
                                                        + 性能日志模式
                                                        + 近期 git log
"把磨皮算法升级一下"         →   [Refactor]  →   加载 beauty-engine 模块
                                                        + 算法文档
                                                        + 基准测试数据
"这个按钮位置不对"           →   [UIAdjust]  →   加载 Compose 布局代码
                                                        + 设计规范
"准备发版"                   →   [Release]   →   加载版本管理流程
                                                        + CHANGELOG
                                                        + 待办清单
```

#### 3.1.2 歧义消除机制

**自动消除（无需人工）**：
- 术语对齐：将口语化表达映射到项目术语表（如"磨皮"→`smoothSkinStrength`）
- 范围推断：根据修改文件自动推断影响模块
- 默认值填充：缺失参数使用模块 AGENTS.md 中的默认值

**需要人工确认（保守执行触发）**：
- 跨模块影响 > 2 个模块
- 涉及全局红线（PRIVACY / PERF / I18N）
- 删除功能或修改公共 API
- 自然语言存在多种技术解读

#### 3.1.3 落地工具

```yaml
# .kimi/skills/intent-router/SKILL.md（建议新建）
- 项目术语表: 维护口语→代码术语映射
- 意图分类器: 基于关键词+文件变更历史的规则分类
- 上下文加载器: 根据意图自动读取相关 AGENTS.md 和源码
```

### Layer 2: 规划闭环 (Plan)

**目标**：将任务描述转化为可原子执行、可验证的实施计划。

#### 3.2.1 自动化影响分析

```kotlin
// 伪代码：影响分析引擎
fun analyzeImpact(task: Task): ImpactReport {
    // 1. 代码依赖图分析（基于 AST/Gradle 依赖）
    val affectedModules = dependencyGraph.findAffected(task.targetFiles)
    
    // 2. 接口兼容性检查
    val apiBreaks = apiCompatibility.check(task.proposedChanges)
    
    // 3. 性能影响预测
    val perfRisk = perfModel.predict(task.description)
    
    // 4. 测试覆盖检查
    val testGaps = coverageAnalyzer.findGaps(task.targetFiles)
    
    return ImpactReport(affectedModules, apiBreaks, perfRisk, testGaps)
}
```

**实现路径**：
- **短期**：基于文件路径的规则匹配（如修改 `beauty-engine/` → 影响相机+相册+编辑）
- **中期**：集成 Gradle 依赖报告 + Kotlin AST 分析脚本
- **长期**：训练项目特定的轻量级影响预测模型

#### 3.2.2 任务拆解模板

| 任务类型 | 自动拆解模板 | 验收标准 |
|----------|-------------|----------|
| **Feature** | ① 数据层 → ② Domain 层 → ③ UI 层 → ④ 测试 → ⑤ 文档 | 功能可用 + 测试覆盖 > 80% |
| **BugFix** | ① 复现 → ② 根因定位 → ③ 修复 → ④ 回归测试 → ⑤ 根因锁定测试 | 复现步骤通过 + 新增测试锁定分支 |
| **Refactor** | ① 接口兼容性检查 → ② 逐步替换 → ③ 编译验证 → ④ 行为一致性测试 | 所有测试通过 + 无行为变更 |
| **UIAdjust** | ① 布局修改 → ② 预览截图 → ③ 多分辨率验证 → ④ 无障碍检查 | 截图对比通过 |
| **Release** | ① 版本号更新 → ② CHANGELOG → ③ 全量回归 → ④ 构建签名包 | regression-test.sh 全绿 |

#### 3.2.3 规划确认策略

```
自动执行模式（默认）:
    影响模块 ≤ 2 且 无 API 破坏 且 性能风险低
    → 自动生成计划 → 直接执行

保守执行模式:
    影响模块 > 2 或 存在 API 破坏 或 性能风险中/高
    → 生成计划 → 展示影响报告 → 等待用户确认
```

### Layer 3: 编码闭环 (Code)

**目标**：AI 自主完成编码、编译修复、直到验证通过。

#### 3.3.1 增量编译优化

**问题**：Android 大型项目 `./gradlew assembleDebug` 耗时 2-5min，AI 修复循环效率极低。

**解决方案**：

```bash
# 1. 模块级编译（只编修改的模块）
./gradlew :app:compileDebugKotlin          # 仅编译 Kotlin，~30s
./gradlew :beauty-engine:compileDebugKotlin # 仅编修改模块，~15s

# 2. 编译缓存守护
./gradlew --build-cache --configure-on-demand :app:assembleDebug

# 3. 快速失败检查（不等待完整构建）
./gradlew :app:ktlintCheck                  # ~10s
./gradlew :app:compileDebugKotlin           # ~30s
./gradlew :app:assembleDebug               # ~2-5min（最终验证）
```

**编码闭环流程**：
```
代码修改
    ↓
快速编译检查 (ktlint + compileDebugKotlin)  [目标: < 1min]
    ↓ 失败
错误分类 → 自动修复策略
    ├── 语法错误     → 直接修复语法
    ├── 类型不匹配   → 推断正确类型并修复
    ├── 导入缺失     → 自动添加 import
    ├── 接口变更     → 级联修改所有调用点
    └── 未知错误     → 读取错误日志 → 定位文件 → 分析上下文 → 修复
    ↓ 成功
完整编译验证 (assembleDebug)               [目标: 1 次通过]
    ↓
单元测试运行 (testDebugUnitTest)            [目标: 全部通过]
```

#### 3.3.2 自愈预算与策略

当前规则：RD 单任务最多自愈 2 次。

**增强策略**：

```
第 1 次失败:
    → 读取错误日志 → 精确定位 → 修复 → 快速编译验证
    
第 2 次失败:
    → 分析失败模式
    ├── 与第 1 次相同问题   → 换修复策略（如从"修改A"改为"修改B"）
    ├── 新问题（级联错误）   → 扩大修复范围
    └── 无法理解            → 读取相关 Skill → 获取领域知识 → 再修复
    
第 3 次失败（超限）:
    → 停止自动修复
    → 生成诊断报告（错误日志 + 已尝试方案 + 建议方向）
    → 上报用户，给出 2 个备选方案
```

#### 3.3.3 代码生成约束

**强制检查项（编码时自动执行）**：
- [ ] 是否符合模块 AGENTS.md 的架构约束
- [ ] 是否涉及 I18N（新增字符串→同步 3 个 strings.xml）
- [ ] 是否涉及隐私（人脸/OCR/分类→确认本地处理）
- [ ] 日志标签是否为 `PicMe:[ModuleName]` 格式
- [ ] 是否使用显式 Lambda 参数名（禁止 `it`）
- [ ] 状态管理是否使用 Sealed Class

### Layer 4: 验证闭环 (Validate)

**目标**：代码编译通过后，自动完成从单元测试到设备端回归的完整验证。

#### 3.4.1 三级验证金字塔

```
                    ▲
                   / \
                  / E2E \      端到端回归 (regression-test.sh)
                 /─────────\    ~5-10min, 连接设备时执行
                /           \
               / Integration \  集成测试 (connectedAndroidTest)
              /───────────────\  ~3-5min, 有 instrumented test 时
             /                 \
            /   Unit + Static   \ 单元测试 + 静态分析 (ai-gate.sh)
           /─────────────────────\ ~1-2min, 每次必执行
```

#### 3.4.2 设备端验证增强

**当前能力**：截屏 + 日志收集 + 人工判断

**增强方向**：

| 验证项 | 当前状态 | 目标状态 | 技术方案 |
|--------|----------|----------|----------|
| **黑屏检测** | Python 脚本分析亮度 | ✅ 已自动 | 保持 |
| **人脸检测** | Python + OpenCV | ✅ 已自动 | 保持 |
| **UI 对比** | 人眼对比截图 | 🔄 待增强 | 像素级 diff + 阈值告警 |
| **布局偏移** | 人工发现 | 🔄 待增强 | OCR 提取坐标 → 与基准对比 |
| **性能基线** | 人工查看日志 | 🔄 待增强 | 自动提取 FPS/耗时 → 与基线对比 |
| **Crash 检测** | 人工查看 logcat | 🔄 待增强 | 自动扫描 `FATAL EXCEPTION` |

**UI 回归自动化方案**：
```bash
# 1. 建立基准截图库
scripts/auto_test_output/baseline/
    ├── camera_preview.png
    ├── gallery_grid.png
    ├── editor_panel.png
    └── ...

# 2. 每次验证时对比
python3 scripts/compare_screenshot.py \
    --baseline baseline/camera_preview.png \
    --current $OUTPUT_DIR/screen_startup.png \
    --threshold 0.95
    
# 3. 输出 diff 热力图
# 相似度 < 95% → 告警（可能是 UI 变更或渲染异常）
```

#### 3.4.3 验证结果决策树

```
三级验证全部通过
    → 自动生成验证报告 → 进入 Layer 5

单元测试通过，但设备验证发现问题
    ├── 黑屏/崩溃      → 回退到 Layer 3 修复渲染/生命周期问题
    ├── 性能超标       → 回退到 Layer 3 优化算法
    └── UI 偏移        → 回退到 Layer 3 调整布局

单元测试失败
    ├── 新功能无测试   → 自动生成测试骨架（基于修改的 public 方法）
    ├── 既有测试失败   → 分析失败断言 → 修复代码或更新测试
    └──  flaky 测试    → 标记 flaky → 重试 3 次 → 仍失败则修复
```

```
    ├── Bug 修复    → 提示更新 CHANGELOG / 已知问题列表
    └── 重构        → 提示更新架构文档
    ↓
AI 自动生成文档变更草案
    ↓
用户确认（保守执行）或 自动提交（自动执行）
```

## 4. 人机协作边界

### 4.1 全自动（无需人工确认）

| 场景 | 说明 |
|------|------|
| 编译错误自动修复 | 语法、类型、导入等明确错误 |

| 日志收集与报告 | 全自动 |
| 单模块内的纯 UI 调整 | 不影响业务逻辑的布局/样式修改 |

### 4.2 自动执行但需报告（人可事后审查）

| 场景 | 说明 |
|------|------|

| 公共 API 变更 | 自动检查兼容性并报告 |
| 性能敏感代码修改 | 自动对比基线并报告差异 |
| 文档自动同步 | AI 生成草案，自动提交但记录变更 |

### 4.3 必须人工确认（保守执行强制触发）

| 场景 | 说明 |
|------|------|
| 删除功能/文件 | 不可逆操作 |
| 修改全局红线相关代码 | PRIVACY / PERF / I18N |
| 数据库 Schema 变更 | 涉及数据迁移 |
| 第三方 SDK 引入/升级 | 合规与授权审查 |
| 自然语言需求歧义 | 存在多种技术解读 |
| 连续自愈超限 | 第 3 次失败时上报备选方案 |

## 7. 风险与应对

| 风险 | 影响 | 应对策略 |
|------|------|----------|
| **AI 过度自信导致错误修复** | 高 | 自愈超限强制上报；关键修改必须人类确认 |
| **设备碎片化导致验证不准** | 中 | 多设备轮询测试；以主流分辨率为主 |

| **文档自动更新质量低** | 低 | 保守执行模式下人工审核；自动执行模式下记录变更便于回溯 |
| **上下文窗口溢出** | 中 | 增量读取代码（只读修改文件+接口文件）；使用 Grep 精确定位 |
