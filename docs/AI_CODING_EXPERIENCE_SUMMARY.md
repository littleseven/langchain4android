# PicMe AI Coding 经验与教训总结

**文档版本**: 2.2  
**最后更新**: 2026-05-05  
**维护者**: [CO] 协调者 / [RD] 全栈工程师

---

## 1. 引言：AI 原生的研发探索

PicMe 项目通过引入 **AI Coding（AI 协作研发）** 模式，在 47 天内完成了传统模式下需数月才能交付的商业级相机应用。本文档总结了从“辅助编程”向“AI 原生协作”演进过程中的成功经验、深刻教训及能力边界。

## 2. 核心成功范式 (Success Paradigms)

### 2.1 多角色串行流转工作流
*   **机制**：建立 `[CO] -> [PM] -> [RD] -> [CR] -> [QA]` 的标准化协作链路。
*   **实践**：通过“触发口令”（如 `自动执行`）启动任务，AI 能够模拟真实团队的协作过程。
*   **价值**：确保了需求的完整性、代码的规范性以及质量验收的闭环。

### 2.2 单一事实来源 (SSOT) 文档体系
*   **架构**：`PRODUCT.md` (目标) -> `FEATURES.md` (交互) -> `模块 AGENTS.md` (实现)。
*   **价值**：为 AI 提供了明确的“唯一真理”，消除了不同文档间的信息冲突，使性能指标和隐私红线等核心要求能准确传导至代码层。

### 2.3 自愈闭环与技能化封装
*   **Self-Heal Loop**：赋予 RD Agent “自愈预算”（单任务最多 2 次尝试），当编译失败时基于日志自动修复。
*   **Skills Integration**：将高频、复杂的任务封装为 Lingma Skills（如 `av-gl-expert`, `image-quality-checker`），实现了专家经验的标准化和可复用。

## 3. 深度复盘：弯路、教训与优化 (Retrospective)

### 3.1 视觉算法的“坐标系迷宫”
*   **痛点**：在处理大眼、瘦脸等变形效果时，团队在 GPU Shader（反向映射）与 CPU Canvas（正向映射）之间反复调试。由于两者在数学公式上的偏移符号相反（`+` vs `-`），AI 曾多次出现“预览是瘦脸，拍照变丰脸”的逻辑冲突。
*   **根因**：AI 缺乏对 NDC、像素坐标及纹理坐标转换的直观物理感知。
*   **对策**：建立统一的 **坐标映射中间层（Coordinate Adapter）**，并在专项文档中固化符号对照表。

### 3.2 UI 调试的“盲目试错”陷阱
*   **痛点**：在修复专业模式曝光滑杆失效问题时，经历了 4 次无效迭代（删除按钮、改样式、调层级、加 padding）才最终定位到状态管理缺失的问题。
*   **根因**：缺乏前置诊断，AI 倾向于直接修改代码而非分析状态流向。
*   **对策**：确立 **“日志驱动开发”** 红线。在修改任何 UI 代码前，必须先通过 `Log.d` 和 Layout Inspector 确认组件的状态流向。

### 3.3 复杂管线的“单体调试”瓶颈
*   **痛点**：早期的 `BeautyRenderer` 将所有渲染逻辑写在一个巨大的 Shader 文件中，导致 AI 在进行局部优化（如调整磨皮算法）时，需要重新编译整个管线，排查错误极其困难。
*   **对策**：实施 **Shader 模块化加载**。将美颜、滤镜、妆容拆解为独立模块，可以让 AI 像调试普通函数一样精准定位图形学问题。

### 3.4 深度学习模型的归一化陷阱
*   **痛点**：在使用 InsightFace 等 ONNX 模型时，AI 容易忽略不同模型对输入图像的归一化要求（如均值、方差、RGB/BGR 顺序），导致检测结果严重偏离。
*   **对策**：建立 **模型配置标准化仓库** (`ModelConfigRegistry`)，将每个模型的元数据和预处理逻辑封装成标准对象。

### 3.5 模型接入的"预处理黑洞"与反复修复
*   **痛点**：在接入 InsightFace 两阶段人脸检测（Det10G + 2D106）的一周时间内，经历了至少 5 轮重大修复：颜色格式从 BGR 修正为 RGB、移除多余的 sigmoid 激活、调整裁剪缩放系数（1.5f → 1.2f）、修正 YUV420→NV21 的 rowStride padding 处理、重构 RetinaFace 的 bbox 解码与 anchor 计算逻辑。每一次修复都意味着大量 Token 被消耗在重新编译、日志诊断和反复验证上。
*   **根因**：AI 缺乏对 ONNX 模型输入输出契约的"精确感知"。当官方 Python 实现使用 `cv2.dnn.blobFromImage(swapRB=True)` 时，AI 在移植到 Kotlin 时默认沿用了 BGR 假设；当模型输出已经是概率值时，AI 习惯性地套用了 sigmoid。这些"经验主义"错误在视觉算法领域代价极高。
*   **对策**：建立 **模型接入 checklist**，在接入任何新模型前必须明确：颜色通道顺序、归一化参数（均值/方差）、输入尺寸、输出格式（概率/logits/偏移量）、后处理公式（NMS/解码方式）。所有假设必须对照官方 reference implementation 逐行验证。

### 3.6 坐标系对齐的"幽灵复发"
*   **痛点**：在移除 GPUPixel 并统一为单引擎后，前置/后置摄像头的人脸关键点坐标对齐问题再次爆发。`mapNormalizedToUv` 中的双重镜像、`BeautyRenderer.inverseTextureMatrix` 与 `pass_copy.glsl` 前置镜像的多层耦合，导致唇色/腮红/瘦脸/大眼效果在前置摄像头出现错位。修复涉及 9 个文件的联动修改，包括 Adapter 层、Renderer 层和 Shader 层。
*   **根因**：尽管项目已有"坐标映射中间层"（见 3.1），但在架构重构（移除 GPUPixel、迁移单引擎）时，坐标系处理逻辑被分散在多个层级（Detector→Adapter→Renderer→Shader），任何一层的变更都会产生级联影响。AI 在重构时倾向于"局部修复"，缺乏对跨层级坐标传导的全局视角。
*   **对策**：确立 **架构变更时的坐标系审计红线**。任何涉及渲染管线、相机预览或人脸检测的架构调整，必须先绘制"数据流-坐标变换图"，明确每一层的输入坐标系、输出坐标系和变换矩阵，经人工审核后再编码。

### 3.7 大模块移除的"文档震荡"
*   **痛点**：移除 GPUPixel 模块（460+ 文件、15万+ 行代码）后，为保持文档与代码一致，进行了多轮文档同步修正：统一 FEATURES.md 与 PRODUCT.md 的引擎架构描述、更新 ADR-001/002、修正 README 技术路线、删除移植计划文档等。文档修改消耗了大量 Token，且出现反复（如"统一文档术语"后又发现遗漏）。
*   **根因**：大模块移除是"结构性变更"，影响面广，但 AI 在移除代码时难以一次性识别所有受影响的文档节点。文档间的引用关系（如 ADR 引用移植计划、README 引用双引擎架构）形成了隐形依赖网。
*   **对策**：建立 **架构变更影响面矩阵**。在删除或重构核心模块前，先通过 grep 扫描所有文档中对该模块的引用，生成"文档-代码一致性检查清单"，按清单批量更新，避免反复修补。

### 3.8 GPU 离屏渲染的"线程与状态"泥潭
*   **痛点**：在实现拍照 GPU 化离屏渲染功能后，连续遭遇多轮严重故障：GPU 拍照黑屏、EGL 上下文在线程池中的绑定错误、拍照与预览效果不一致、FBO 解绑时机不当。修复涉及 `PhotoProcessor` 线程切换、`BeautyRenderer` 离屏 Surface 生命周期、EGLContext 的 makeCurrent/release 时序，共经历 5+ 次提交才彻底稳定。
*   **根因**：AI 对 EGL 状态机（Context、Surface、Display 的三元绑定关系）缺乏深度理解，倾向于"在症状处打补丁"（如在拍照前强制创建新上下文），而非理清"谁创建、谁绑定、谁释放"的完整状态流转。线程池环境下的 EGL 上下文复用更是超出了 AI 的常规经验范围。
*   **对策**：建立 **离屏渲染状态机规范**。任何涉及 EGL 的操作必须遵循：① 单线程创建与销毁 ② `makeCurrent`/`release` 成对出现 ③ 拍照/预览共享上下文时必须显式传递纹理 ID 而非重复渲染 ④ 在 `finally` 块中确保 FBO 和 Surface 解绑。关键 EGL 操作必须添加 `PicMe:EGL` 日志标签以便追踪状态流转。

### 3.9 预览与拍照效果不一致的"双路径"困境
*   **痛点**：早期拍照后处理走 CPU Canvas 路径（`ImageProcessor`），预览走 GPU Shader 路径（`BeautyRenderer`），两者在磨皮强度、色彩空间、人脸变形算法上存在系统性差异，导致"预览好看，拍照变样"。为统一路径，启动 GPU 离屏渲染拍照方案后，又连续遭遇：Bitmap 纹理上传格式错误（应使用 `GL_TEXTURE_2D` 而非 `GL_TEXTURE_EXTERNAL_OES`）、拍照上下文缺少人脸数据、Shader 参数在预览/拍照间未同步、PBO 读取图像上下翻转等问题。从发现问题到完全解决，经历了 4+ 轮迭代。
*   **根因**：AI 将"预览渲染"与"拍照处理"视为两个独立问题分别设计路径，未在架构层面统一 Shader 管线。当需要统一时，又忽视了两种场景的关键差异：预览输入是 OES 外部纹理（相机直连），拍照输入是 2D 纹理（Bitmap 上传）；预览人脸数据来自实时检测流，拍照人脸数据需要缓存或重新检测；Canvas 与 Shader 的色彩空间转换逻辑完全不同。
*   **对策**：确立 **"单管线多入口"架构原则**。所有图像处理必须复用同一套 Shader 管线，仅通过输入纹理类型（OES/2D）和 viewport 尺寸区分预览与拍照。在 `BeautyRenderer` 中显式支持两种输入模式，禁止为拍照单独维护一套渲染逻辑。拍照前必须校验人脸数据缓存是否命中，未命中时应触发同步检测或降级为无美型效果。

## 4. AI Coding 的能力边界 (Boundaries of AI)

在 PicMe 项目中，以下工作被证明**不适合完全交给 AI**：

1.  **跨坐标系空间映射**：涉及底层图形学原理的复杂转换，需人类专家定义标准。
2.  **深度根因定位**：无日志指引下的复杂 Bug 诊断，AI 易陷入盲目搜索。
3.  **架构战略决策**：长期技术债务权衡与模块边界划分，需人类宏观视野。
4.  **审美与体验微调**：情感化设计与品牌调性打磨，AI 难以捕捉主观美感。
5.  **非标资源集成**：缺乏文档的老旧 SDK 接入，需人类实验获取第一手数据。
6.  **模型参考实现的精确翻译**：AI 在将官方 Python/C++ reference 翻译为 Kotlin 时，容易在颜色格式、归一化参数、激活函数等"微观差异"上出错，需人类逐行对照验证。

## 8. 多工具协同实践 (Multi-Tool Synergy)

在 PicMe 项目中，我们同时采用了 **Lingma**、**Kimi-cli**、**Gemini** 和 **CodeBuddy**，形成了“主从协作+冗余校验”的模式。

### 8.1 角色定位
*   **Lingma (主执行者)**：深度集成 IDE，承担 70% 的日常编码、文件跳转和基础调试。
*   **Kimi-cli (长文本专家)**：利用超长上下文处理多层级文档审计及大规模 Logcat 日志分析。
*   **Gemini (逻辑顾问)**：负责复杂图形学原理（如 Shader 映射）的深度推理与架构备选方案。
*   **CodeBuddy (独立审查者)**：提供不受 IDE 上下文干扰的“冷启动”视角，用于边界条件推演。

### 8.2 典型教训案例 (Lessons Learned Cases)

#### Case 1: “上下文割裂”导致的重复劳动
*   **场景**：在调整 InsightFace 归一化参数时，从 Lingma 切换到 Kimi-cli，但未同步 `ImageUtils.kt` 等依赖文件。
*   **后果**：Kimi 基于通用经验给出错误建议，导致“改 A 坏 B”。
*   **对策**：建立 `CONTEXT.md` 机制，切换工具前必须汇总相关依赖代码与调试日志。

#### Case 2: “指令风格错位”引发的低效沟通
*   **场景**：对 Kimi-cli 使用针对 Lingma 的简短指令（如“修复这个空指针”）处理 5000 行日志。
*   **后果**：Kimi 输出泛泛而谈的可能性，未定位根因。
*   **对策**：建立分工具 Prompt 模板。对 Kimi/Gemini 强制使用“背景-任务-约束-输出格式”四段式指令。

#### Case 3: “冗余校验”带来的决策瘫痪
*   **场景**：Gemini 与 CodeBuddy 在坐标映射方案上产生分歧（NDC 统一 vs CPU 像素转换）。
*   **后果**：开发者在两种方案间犹豫两天，浪费大量 Token 编写对比 Demo。
*   **对策**：确立“主从仲裁”原则。日常以 Lingma 为准；底层分歧以“数学推导更严谨”的一方为最终裁决。

#### Case 4: “责任归属模糊”导致的 Bug 遗漏
*   **场景**：Lingma 忽略了 CodeBuddy 提出的状态同步风险，双方自行“协商”后未修复。
*   **后果**：边界 Bug 留到真机测试阶段才被发现。
*   **对策**：AI 互审不能替代人类终验。意见不一致时必须触发人工介入（Human-in-the-loop）。

#### Case 5: "长上下文过载"导致的模型接入低效
*   **场景**：在接入 InsightFace Det10G 时，将 410 行新检测器代码 + 277 行 Manager 重构代码 + 模型官方 Python 参考实现一次性提交给 AI 分析。
*   **后果**：AI 在处理超长代码片段时，对关键细节（如 `swapRB=True` 的含义、输出是否经过 sigmoid）产生"选择性失明"，连续给出错误的颜色格式和激活函数假设，导致 3 轮无效修复，浪费大量 Token。
*   **对策**：**模型接入时采用"分而治之"策略**。先提交模型配置元数据（输入尺寸、归一化参数、输出头定义），确认理解无误后再提交具体实现代码；遇到精度问题时，优先提供最小可复现的输入/输出对比（如 Python reference vs Kotlin output），而非完整文件。
