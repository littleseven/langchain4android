## 1. 文档定位与目的

### 1.1 什么是 AGENTS.md

AGENTS.md 是 PicMe 项目的**技术实现规范文档**,为每个模块提供详细的技术指导、实现标准和最佳实践。

### 1.2 文档层级关系

```
PRODUCT.md (产品需求规格说明书 - What)
    ↓ 引用
FEATURES.md (功能交互细节规范 - How)
    ↓ 指导
AGENTS.md (技术实现规范 - Implementation)

AI_TOOLS.md (AI 工具配置索引)
DEVELOPMENT.md (通用开发指南)
```

> **新增文档说明**（2026-05）：
> - `AI_TOOLS.md`：统一索引所有 AI 辅助工具的配置位置（Qoder / kimi-cli / OpenClaw / Cursor），与主文档体系并列，不作为产品或技术规范的上下级。
> - `DEVELOPMENT.md`：通用开发环境配置与命令速查，面向人类开发者与所有 AI 工具。
>
> **状态标注规范**（2026-05-24 新增）：
> 文档中涉及功能/组件/决策时，统一使用以下标注：
> - ✅ **已落地** — 代码已实现，生产可用
> - 🔄 **部分实现** — 代码有骨架或核心链路已通，待完善
> - ⏳ **规划中** — 有设计文档，代码尚未开始
> - ❌ **已废弃** — 曾计划或曾实现，已移除
> - 📝 **待验证** — 文档描述与代码可能不一致，需人工核对

### 1.3 核心原则

- **单一职责**: 每个 AGENTS.md 只聚焦一个模块的技术实现
- **格式统一**: 所有模块遵循相同的 5 章结构
- **可执行性**: 包含具体的检查清单和执行规约
- **可追溯性**: 明确对照 PRODUCT.md 和 FEATURES.md 的产品要求

### 1.4 顶层 AGENTS 与模块 AGENTS 的边界（2026-04）

**顶层 `AGENTS.md` 只保留**：
- 角色与协作机制（CO/PM/RD/CR/QA）
- 文档治理规则（SSOT、更新顺序、引用链）
- 全局红线（PRIVACY / PERF / I18N）
- 跨模块工程基线（架构、日志、导入、状态管理）
- Self-Heal 执行流程与统一验收口径

**禁止放在顶层 `AGENTS.md` 的内容**：
- 业务功能细节（应放 `PRODUCT.md`、`docs/FEATURES.md`）
- 模块级实现策略、代码片段、参数表（应放模块 `AGENTS.md`）
- 专项技术长文（应放 `docs/*_TECH_SPEC.md`）

**内容分流原则**：
- `What`（目标/指标）-> `PRODUCT.md`
- `How`（交互/体验）-> `docs/FEATURES.md`
- `Implementation`（代码/架构/检查清单）-> 模块 `AGENTS.md`
- `Deep Dive`（专项机制）-> 技术规范文档
- `AI Config`（工具配置/兼容性）-> `AI_TOOLS.md`
- `Dev Guide`（环境/构建/调试）-> `DEVELOPMENT.md`

## 2. 标准文档结构

### 2.1 文件头规范

```markdown
# [模块名] 技术实现规范 ([English Name])

> **边界声明（Boundary Statement）**
> - 本文档仅承载本模块的实现细节（架构、代码约束、检查清单）。
> - 产品目标与验收口径以 `PRODUCT.md` 为准；交互流程与体验规则以 `docs/FEATURES.md` 为准。
> - 顶层治理规则（角色协作、全局红线、文档流程）以根目录 `AGENTS.md` 为准。
> - 禁止将模块级实现细节回填到顶层 `AGENTS.md`；跨模块或专项技术内容应下沉到对应模块文档或 `docs/*_TECH_SPEC.md`。

**模块定位**: [一句话描述模块的核心职责]

**主要维护者**: [RD] 全栈工程师

**阅读对象**: RD、AI Agent
```

**要求**:
- 标题必须使用 `# [模块名] 技术实现规范` 格式
- 标题下方必须紧跟统一的“边界声明（Boundary Statement）”模板
- 英文名称放在括号内作为副标题
- 模块定位应简洁明了 (不超过 50 字)
- 明确标注维护者和阅读对象

### 2.2 五章标准结构

**所有 AGENTS.md 必须严格遵循以下章节顺序**:

```markdown
## 1. 核心产品逻辑 (Core Product Logic)
## 2. 技术实现规范 (Technical Implementation)
## 3. Agent 执行规约 (Execution Rules)
## 4. 常见陷阱检查清单 (Checklist)
## 5. 与产品文档对照 (Product Alignment)
```

**严禁**:
- ❌ 调整章节顺序
- ❌ 删除任一章节
- ❌ 添加额外的主章节 (可使用子章节)

## 3. 各章节编写细则

### 3.1 第 1 章：核心产品逻辑 (Core Product Logic)

**目的**: 定义该模块必须遵守的核心产品原则和技术约束。

**格式**:
```markdown
## 1. 核心产品逻辑 (Core Product Logic)

- **[LABEL_1] 原则名称 1**: 具体描述
- **[LABEL_2] 原则名称 2**: 具体描述
- **[LABEL_3] 原则名称 3**: 具体描述
```

**常用标签**:
| 标签 | 含义 | 适用场景 |
|------|------|---------|
| `[PERF]` | 性能要求 | 启动速度、帧率、响应时间 |
| `[PRIVACY]` | 隐私保护 | 本地处理、权限最小化 |
| `[I18N]` | 国际化 | 多语言支持、文案提取 |
| `[DESIGN_CONSISTENCY]` | 设计一致性 | HyperOS 视觉风格 |
| `[FEEDBACK]` | 用户反馈 | 触感、音效、动画 |
| `[LOCAL]` | 本地存储 | 零云端、DataStore |
| `[TYPE_SAFE]` | 类型安全 | Sealed Class、DataStore |
| `[ACCURACY]` | 精度要求 | 人脸分组、重复检测 |
| `[OFFLINE]` | 离线可用 | ML Kit、无网络依赖 |
| `[RESPONSE]` | 响应速度 | OCR 秒级响应 |

**示例** (Camera 模块):
```markdown
## 1. 核心产品逻辑 (Core Product Logic)

- **[PERF] 零延迟拍摄**: 快门延迟 < 50ms，使用 `CAPTURE_MODE_MINIMIZE_LATENCY` 模式
- **[FEEDBACK] 三位一体反馈**: 触感 + 音效 + 黑场必须同步触发 (50ms 内)
- **[OFFLINE] OCR 本地识别**: 使用 ML Kit 离线引擎，严禁云端处理
- **[PRIVACY] 权限最小化**: 仅在需要时申请相机和存储权限，提供降级方案
```

**要点**:
- 每条原则控制在 1-2 行
- 必须包含量化的指标 (如 < 50ms)
- 优先使用项目统一的标签

### 3.2 第 2 章：技术实现规范 (Technical Implementation)

**目的**: 提供详细的技术实现指南、代码示例和最佳实践。

**格式**:
```markdown
## 2. 技术实现规范 (Technical Implementation)

### 2.1 [子模块 A]
- 技术规范 1
- 技术规范 2
- 代码示例 (可选)

### 2.2 [子模块 B]
- 技术规范 1
- 技术规范 2
```

**子章节划分建议**:

根据不同模块的特点划分子章节:

**Core 模块**:
```markdown
### 2.1 Design System 实现
### 2.2 图像处理核心
### 2.3 工具类与扩展函数
### 2.4 依赖注入配置
```

**Data 模块**:
```markdown
### 2.1 Room 数据库规范
### 2.2 DataStore 偏好设置
### 2.3 Repository 层职责
```

**Feature 模块** (Camera/Gallery/Settings):
```markdown
### 2.1 [核心技术] 配置要求
### 2.2 [业务逻辑] 实现细节
### 2.3 [集成功能] 接入规范
### 2.4 性能优化关键点
```

**代码示例规范**:
```kotlin
// ✅ 好的示例：包含注释、符合规范
// Repository 层 - 仅加载元数据，严禁加载大图
@Query("SELECT * FROM media_assets ORDER BY date_taken DESC")
fun getAllMediaMetadata(): Flow<List<MediaAsset>>

// ❌ 差的示例：缺少注释、不规范
fun getData() = query("SELECT * FROM table")
```

**要点**:
- 使用 `### 2.x` 组织子模块
- 关键技术点使用**粗体**标注
- 代码示例必须包含注释说明
- 避免大段代码，保持简洁

### 3.3 第 3 章：Agent 执行规约 (Execution Rules)

**目的**: 列出 AI Agent 在修改该模块时必须遵守的执行规则。

**格式**:
```markdown
## 3. Agent 执行规约 (Execution Rules)

- **关键词 1**: 具体规则描述
- **关键词 2**: 具体规则描述
- **关键词 3**: 具体规则描述
```

**规则分类**:

**通用规则** (适用于所有模块):
- **I18N**: 所有用户可见字符串必须提取到 strings.xml
- **日志**: 必须使用正确的 Tag 格式 (PicMe:Module)
- **线程**: 耗时操作必须在后台线程执行
- **异常**: 必须处理可能的异常情况

**模块特定规则**:
- **Camera**: 拍摄操作必须在后台线程保存照片
- **Gallery**: LruCache 必须设置合理上限
- **Data**: 修改 Entity 后必须更新 Database 版本号
- **DI**: 新增 Repository 后必须更新 AppContainer.kt

**示例**:
```markdown
## 3. Agent 执行规约 (Execution Rules)

- **拍摄操作**: 必须在后台线程保存照片，避免阻塞 UI
- **OCR 管理**: OCR 引擎在空闲 30s 后必须释放，避免内存泄漏
- **夜景模式**: 必须正确应用曝光补偿 (+1.0 to +2.0)
- **权限处理**: 权限被拒绝时必须提供降级方案和引导设置
- **I18N**: 所有用户可见字符串必须提取到 strings.xml
```

**要点**:
- 每条规则以**关键词**开头
- 描述应具体、可执行
- 避免模糊的表述 (如"应该"、"尽量")
- 使用"必须"、"严禁"等强制性词汇

### 3.4 第 4 章：常见陷阱检查清单 (Checklist)

**目的**: 提供开发和 Code Review 时的检查项列表。

**格式**:
```markdown
## 4. 常见陷阱检查清单 (Checklist)

- [ ] 疑问句形式的检查项 1? (解释说明)
- [ ] 疑问句形式的检查项 2? (解释说明)
- [ ] 疑问句形式的检查项 3? (解释说明)
```

**检查项来源**:
1. 历史踩坑经验
2. Code Review 常见问题
3. 性能优化关键点
4. 产品规范要求

**示例** (Gallery 模块):
```markdown
## 4. 常见陷阱检查清单 (Checklist)

- [ ] 是否在 UI 线程中加载了大图？(必须使用 Dispatchers.IO)
- [ ] LruCache 是否设置了合理的上限？(避免 OOM)
- [ ] Face Detection 是否在后台线程执行？(ML Kit 很耗时)
- [ ] OCR 引擎是否在使用后释放？(避免内存泄漏)
- [ ] 滚动时是否频繁创建对象？(应在构造函数中初始化)
- [ ] Flow 是否正确使用了 `stateIn`?(避免重复订阅)
- [ ] 图片加载是否处理了异常？(避免崩溃)
- [ ] 长按触发 OCR 时是否有防抖？(避免重复触发)
```

**要点**:
- 必须使用 `- [ ]` Markdown 复选框格式
- 必须是**疑问句**形式
- 括号内提供简要解释或解决方案
- 通常包含 6-10 条检查项
- 按重要性和频率排序

### 3.5 第 5 章：与产品文档对照 (Product Alignment)

**目的**: 建立技术实现与产品需求的可追溯关系。

**格式**:
```markdown
## 5. 与产品文档对照 (Product Alignment)

**必须满足的产品指标**:
- ✅ 产品指标 1 → 技术方案
- ✅ 产品指标 2 → 技术方案

**技术决策记录**:
- 选择 A 而非 B: 原因 1、原因 2、原因 3
```

**产品指标部分**:
- 列出 PRODUCT.md 或 FEATURES.md 中定义的产品指标
- 说明对应的技术实现方案
- 使用 `✅ 指标 → 方案` 格式

**技术决策记录部分**:
- 记录重要的技术选型决策
- 说明选择某个方案的原因
- 可以对比其他备选方案

**示例** (Settings 模块):
```markdown
## 5. 与产品文档对照 (Product Alignment)

**必须满足的产品指标**:
- ✅ 零云端 → 所有设置本地存储，不申请网络权限
- ✅ 多语言支持 → 英文、简体中文、繁体中文三语齐全
- ✅ 权限透明 → 明确告知用途，提供降级方案

**技术决策记录**:
- 选择 DataStore 而非 SharedPreferences: 类型安全、支持 Flow、无主线程阻塞
- 使用 Sealed Class 表示设置项：编译器检查 exhaustive when，避免遗漏
- Combine 多个 Flow: 减少订阅次数，提升性能
```

**要点**:
- 产品指标必须与 PRODUCT.md/FEATURES.md 对齐
- 技术决策应说明权衡过程
- 避免过于简略 (至少 3 条指标 +3 条决策)

### 4.4 顶层 AGENTS 瘦身流程（新增）

当顶层 `AGENTS.md` 出现“细节过载”时，按以下步骤治理：
1. **识别超域内容**：标记业务细节、模块实现细节、专项技术细节。
2. **执行分流**：
   - 业务目标与验收口径 -> `PRODUCT.md`

   - 实现细节与检查清单 -> 模块 `AGENTS.md`
   - 长链路技术方案 -> `docs/*_TECH_SPEC.md`
3. **保留治理骨架**：顶层文档仅保留角色、红线、流程、审计清单。
4. **双向链接补齐**：在来源与去向文档都加“见 xxx 文档”指引。
5. **一致性复核**：确保三层文档术语、指标、顺序一致。

## 5. 常见错误与反模式

### ❌ 错误 1: 章节缺失或顺序混乱

```markdown
## 1. 核心产品逻辑
## 3. 常见陷阱检查清单  # ❌ 缺少第 2、3 章
## 2. 技术实现规范      # ❌ 顺序错误
```

**正确做法**: 严格遵循 5 章标准结构，顺序不可调整。

### ❌ 错误 2: 核心逻辑缺少量化指标

```markdown
- **[PERF] 性能优化**: 提升加载速度  # ❌ 不够具体
```

**正确做法**:
```markdown
- **[PERF] 冷启动 < 500ms**: 分阶段初始化，懒加载 AI 模型
```

### ❌ 错误 3: 技术实现只有理论没有示例

```markdown
### 2.1 图片加载
必须使用 Coil 框架加载图片，配置缓存策略。  # ❌ 太抽象
```

**正确做法**:
```markdown
### 2.1 图片加载
**Coil 全局配置**:
- 内存缓存：占可用内存 25%，使用 MemoryCache.Builder
- 磁盘缓存：占磁盘空间 2%，目录为 `cache/image_cache`

```kotlin
ImageLoader(context)
    .memoryCache {
        MemoryCache.Builder(context)
            .maxSizePercent(25)
            .build()
    }
    .diskCache {
        DiskCache.Builder()
            .directory(cacheDir.resolve("image_cache"))
            .maxSizePercent(2)
            .build()
    }
    .build()
```
```

### ❌ 错误 4: 执行规约模糊不清

```markdown
- **性能**: 尽量优化代码性能  # ❌ 无法执行
```

**正确做法**:
```markdown
- **图片加载**: 必须在后台线程 (Dispatchers.IO) 加载大图，严禁在 UI 线程
```

### ❌ 错误 5: 检查清单不是疑问句

```markdown
- [ ] 图片加载的线程管理  # ❌ 这是名词短语
```

**正确做法**:
```markdown
- [ ] 是否在 UI 线程中加载了大图？(必须使用 Dispatchers.IO)
```

### ❌ 错误 6: 产品对照过于简略

```markdown
**必须满足的产品指标**:
- ✅ 性能优化
- ✅ 用户体验  # ❌ 太抽象，无法追溯
```

**正确做法**:
```markdown
**必须满足的产品指标**:
- ✅ 冷启动 < 500ms → 分阶段初始化，懒加载 AI 模型
- ✅ 120fps 滚动 → LruCache + 预加载 + beyondBoundsPageCount
```
