# PRD：帧同步美妆系统（Frame-Sync Makeup System）

**版本**：1.0  
**状态**：需求冻结，待技术评审  
**优先级**：P0（阻塞后续妆容精细化）  
**负责人**：PM / RD  
**最后更新**：2026-05-11

---

## 1. 背景（Why）

### 1.1 问题定义：美妆与人脸分离

当前大美丽引擎已实现**双缓冲 + 顶点插值**来缓解妆容甩飞问题，但用户侧仍感知到"妆容粘在脸上但慢半拍"的分离感。根本原因：**人脸检测与相机预览帧在时间上未对齐**。

| 维度 | 现状 | 问题 |
|------|------|------|
| **检测帧率** | InsightFace/MediaPipe ~10fps | 远低于渲染帧率 30~60fps |
| **时间对齐** | 无帧 ID / 无时间戳匹配 | 渲染帧使用的是"过去某时刻"的人脸关键点 |
| **空间补偿** | 线性插值（基于时间进度） | 仅缓解突变，无法消除系统性滞后 |
| **缺失感知** | 人脸离开画面后妆容仍残留 | 无严格"检测命中才渲染"机制 |

### 1.2 用户体验影响

- **快速转头/移动**：唇色/腮红明显滞后于实际人脸位置（滞后 ~50~100ms）
- **人脸出画入画**：人脸离开画面后，妆容仍停留在最后一帧位置，持续数帧才消失
- **视频录制**：录制帧率稳定 30fps，但妆容更新只有 10fps，视频中出现"妆容跳变"

### 1.3 竞品对标

| App | 方案 | 效果 |
|-----|------|------|
| 轻颜相机 | 硬件同步 + 预测补偿 | 妆容几乎零滞后 |
| BeautyCam | 严格帧绑定 + 插值 | 快速移动时仍有轻微延迟 |
| 当前 PicMe | 双缓冲 + 时间插值 | 可感知滞后，快移时明显 |

**目标**：达到 BeautyCam 同级水平，快速移动场景下妆容与人脸位移偏差 < 8px（1080p 基准）。

---

## 2. 方案（What & How）

### 2.1 核心策略：帧 ID 绑定 + 时序队列 + 预测补偿

将"异步松散耦合"升级为"准同步帧匹配"：

```
┌──────────────┐      ┌──────────────┐      ┌──────────────┐
│  CameraX     │      │  FrameSync   │      │   Render     │
│  预览帧流     │─────▶│  时序队列     │◀─────│   管线       │
│  (30~60fps)  │      │  (帧ID绑定)   │      │  (30~60fps)  │
└──────────────┘      └──────┬───────┘      └──────────────┘
                             │
┌──────────────┐             │
│  FaceDetect  │─────────────┘
│  人脸检测流   │   检测结果携带帧ID入队
│   (~10fps)   │
└──────────────┘
```

### 2.2 功能需求列表

#### FR-1：全局帧 ID 体系（P0）

- 为每一帧从相机获取的图像分配单调递增的 `FrameId: Long`
- `FrameId` 从 SurfaceTexture 的 `updateTexImage()` 时刻生成，贯穿检测→渲染全链路
- 所有与人脸相关的数据结构必须携带 `FrameId`

#### FR-2：人脸检测帧绑定（P0）

- 人脸检测输入必须是**带 FrameId 的帧数据**，检测结果输出携带相同 `FrameId`
- 检测线程从"异步分析最新帧"改为"消费带 FrameId 的帧队列"
- 队列深度可控（默认 2 帧），超时丢弃（避免检测滞后太久）

#### FR-3：时序对齐队列（P0）

- 渲染线程在渲染前，根据当前帧的 `FrameId` 查询对应的人脸检测结果
- 匹配策略：
  - **精确匹配**：当前 FrameId 正好有检测结果 → 直接使用
  - **最近历史**：无精确匹配，取 <= 当前 FrameId 的最新结果
  - **未来不可用**：不借用未来帧的检测结果（避免反向时间线）

#### FR-4：运动预测补偿（P1）

- 当渲染帧与检测帧存在时间差时，基于历史轨迹进行运动预测
- 预测算法：Kalman Filter 或简单速度外推（二选一，先落地简单方案）
- 预测约束：位移幅度不超过上一帧的 150%，避免过度外推算崩

#### FR-5：严格缺失处理（P0）

- 当当前帧及历史 N 帧内均无检测结果时：
  - 选项 A（默认）：隐藏妆容（`uHasFace = 0`），不渲染任何妆容 Pass
  - 选项 B（可选）：保留最后一帧位置（当前行为，作为降级）
- 切换阈值可配置（默认 N = 3 帧，约 50ms@60fps）

#### FR-6：调试可观测性（P1）

- 调试浮层新增指标：
  - `detectionLatencyMs`：检测滞后时间（当前帧时间 - 检测帧时间）
  - `syncStatus`：精确匹配 / 历史回退 / 预测补偿 / 缺失隐藏
  - `predictedOffsetPx`：预测补偿的像素位移量

### 2.3 非功能性需求

| 指标 | 目标值 | 最差容忍 |
|------|--------|----------|
| 检测-渲染对齐精度 | 偏差 < 8px @1080p | < 16px |
| 预测补偿耗时 | < 0.5ms / 帧 | < 1ms |
| 帧 ID 队列内存占用 | < 50KB | < 200KB |
| 严格缺失处理切换延迟 | < 1 帧（16ms） | < 3 帧 |
| 整体渲染帧率影响 | 无下降 | 下降 < 5% |

## 3. 交互与体验规范

### 3.1 用户可见行为

- **正常情况**：用户无感知，妆容自然贴合
- **快速移动**：妆容平滑跟随，无明显滞后或甩飞
- **人脸出画**：妆容在 1~2 帧内消失，不出现"悬空妆容"
- **人脸入画**：妆容在检测到人脸的第 1 帧立即出现（无淡入延迟）

### 3.2 设置项（开发者模式）

在设置页"开发者选项"中新增：

- **帧同步模式**：
  - `严格对齐`（默认）：精确匹配 + 缺失隐藏
  - `平滑优先`：允许历史回退 + 预测补偿
  - `关闭`：恢复当前双缓冲插值行为（用于对比测试）
- **预测算法**：`速度外推`（默认）/ `Kalman Filter`（实验性）
- **缺失阈值**：1~10 帧可调（默认 3 帧）

## 4. 验收标准

### 4.1 P0 必达标

- [ ] 每一帧渲染均携带 `FrameId`，人脸检测结果携带相同 `FrameId`
- [ ] 时序对齐队列正确实现，无内存泄漏
- [ ] 严格缺失处理：人脸离开画面 3 帧内妆容隐藏
- [ ] 快移场景（转头速度 > 90°/s）妆容偏差 < 16px @1080p
- [ ] 帧率无下降（同设备对比关闭帧同步模式）

### 4.2 P1 期望达标

- [ ] 快移场景妆容偏差 < 8px @1080p
- [ ] 预测补偿算法实现，位移平滑无抖动
- [ ] 调试浮层展示 `detectionLatencyMs` / `syncStatus` / `predictedOffsetPx`

### 4.3 P2 优化项

- [ ] Kalman Filter 预测替换速度外推
- [ ] 检测线程与渲染线程完全解耦，检测帧率自适应提升

## 5. 风险与边界

### 5.1 技术风险

| 风险 | 等级 | 缓解措施 |
|------|------|----------|
| 帧 ID 引入增加链路复杂度 | 中 | 封装为 `FrameSyncManager`，对上层透明 |
| 检测线程队列积压导致延迟 | 中 | 队列深度限制 + 超时丢弃 + 降采样 |
| 预测算法不稳定导致妆容抖动 | 低 | 位移约束 + 可配置关闭预测 |
| 低端机性能开销 | 中 | 预测计算在 CPU 完成，量极轻；支持关闭帧同步 |

### 5.2 兼容性

- **前置/后置摄像头**：均需支持，前置镜像已在 `mapViewNormalizedToUv` 处理
- **视频录制**：帧同步必须覆盖录制链路，确保录制帧与预览帧行为一致
- **拍照后处理**：`PhotoProcessorImpl` 需复用同一套帧同步逻辑（单帧场景退化为有无人脸判断）

### 5.3 对库化的影响

- 帧同步属于**横切能力**，应沉淀在 `beauty-engine` 公共层，不侵入 `egl/` 具体 Pass 实现
- `FaceMakeupPass` 只接收"已同步好"的顶点数据，不关心同步逻辑
- 为未来其他依赖人脸关键点的功能（如美体、背景虚化）提供统一同步基础设施
